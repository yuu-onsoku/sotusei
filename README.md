# README

## 1. サービス概要（※最後に書いてOK）
猫を飼いたいけれど、具体的な費用や生活の変化（物を壊したり・抜け毛など）を把握しきれていない方が猫を飼う前に、お迎え診断機能、覚悟のチェックリスト、で自身の状況を知ることができ。その後コミュニティにて、すでに飼っている人とこれから飼いたい人をつなげる猫と人のコミュニケーションアプリです。

## 2. このアイデアはどこから生まれたか
### 2-1. きっかけとなった体験・感情
- いつ
小学生の頃からねこを飼いたいと思ったからです。

- どんな状況で
小学生の頃ねこのテレビ番組や近所の野良猫などを見てねこを飼いたいと思っていましたが、ペット禁止のマンションで飼えないし当時はあまりねこの病院代やお世話代などの金銭面のことやねこの詳しい生体お世話などわかっておらず、かわいいからという安直な理由で飼いたいと思ってしまっていました。

- どんな気持ちになったか
私自身は小学校3年生くらいにはねこを安直にかわいいだけで飼ってはいけないことを学んだが、
テレビやYouTubeなどで、大人になってもかわいいだけで飼おうとしている人がいるのをしりよく無いと思ったことと、「可愛い」という気持ちだけで飼い始めてしまい、多額の通院費や掃除の大変さに直面して困る人や不幸になってしまうねこちゃんを減らしたいと思ったからです。

ねこが自分勝手な人間に飼われてかわいそうと思ったからということと、

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

年齢設定：
  - 年齢：25〜30代
  - 理由：社会人として自立し始め、生活に癒やしを求めて「本気で飼育を検討し出す時期」で、飼いたいけど何から始めたら良いかわからない人に向けて

  - 生活・状況：一人暮らしで寂しくなり元々ねこが好きで初めて猫を迎えようと悩んでいる人。
  - 理由：このアプリでねこを飼うにはこういうことがあるというのを知ってもらいねこを飼う覚悟を決めてもらう

環境設定：
  - 環境：猫を飼いたい意欲はあるが、費用感や具体的なお世話の工程が漠然としている。
  - 理由：猫種ごとの「お迎えコスト・お世話ガイド」を提供することで、漠然とした不安を「具体的な準備」に変え、ミスマッチによる飼育放棄を防ぎます。

## 7. このアプリで実現すること

### MVPで作る機能
#### 「猫を飼いたい」という気持ちに対し、「今の生活環境で本当に最後まで責任を持てるか？」を判定する。★は難しいか不安 <br> 
 - ユーザー登録/ログイン機能：
   - Deviseによる新規登録・ログイン。診断結果をマイページに保存し、後から見返せるようにします。
 - 「お迎え適性」診断機能（メイン機能）：
   - 生活リズム（不在時間、帰省頻度）、住環境、経済状況（月々の可動予算）に関する（5〜10問程度の質問）。
   - ★　回答に基づき「今はまだ早い」「成猫なら可能」「準備万端」などの判定ロジックを実装。
 - 「猫視点」の診断結果表示：
   - 判定結果とともに、猫のキャラクターから「今の生活だとボクたちは寂しいニャ」といった、誠実かつ愛のある理由説明を表示。
 - 「覚悟」のチェックリスト機能:
   - 抜け毛、夜鳴き、壁の傷など、猫飼育の「大変な現実」を羅列。すべてに納得したかを確認し、自分の「覚悟」を視覚化します。
 -

### 本リリースで作る機能
#### 診断をクリアし、「飼う」と決めたユーザーが、具体的にお迎えの準備を進めるための機能を追加します。
 - ステップアップ型・シミュレーター（お迎え準備編）：
   - ペットショップ初期費用計算： 生体価格、ケージ、トイレ等の初期セット、安心パック（保険等）の合算見積もり。
   - 病院アクセス確認： Google Maps APIと連携し、近隣の動物病院を検索・確認する機能。

 - パーソナライズ・ロードマップ機能：
   - 診断結果に基づき、「脱走防止柵を買う」「貯金目標○万円」など、その人に必要な準備タスクを自動生成。
 - 準備進捗の可視化（進捗マップ）：
   - 準備タスクの完了度をグラフやステップ図で表示。
   - 「準備完了！」のステータスをSNSにシェアできる機能。
 - 「現実」の費用レコーディング機能：
   - 実際に購入した猫用品や、迎えた後の医療費を家計簿のように記録。
   - MVP時のシミュレーション金額と実績を比較できる機能。
 - 猫種別・生涯費用シミュレーター（精密版）：
   - 猫種ごとの平均寿命や「かかりやすい病気」のデータを元に、15〜20年間のトータルコストをグラフ化。
 - Googleアカウント連携：
   - Google Authによる簡単ログイン。
 - RSpecによるテストコード実装：
   - 計算ロジックや診断機能が正しく動くかの自動テスト。

## 8. このアプリの懸念点とその対策
※最低1つは必ず書いてください。

このアプリが**うまくいかない可能性**と、
それに対して**どう向き合おうとしているか**を書いてください。

### 懸念点①
懸念点①：費用の乖離と「想定外」の事態
- 何が問題になりそうか：アプリで算出したシミュレーション金額と、実際の費用が大きくズレてしまうこと。
- なぜそう思うか：生体価格の変動だけでなく、猫の健康状態（持病や突然の事故）によって医療費は青天井になる可能性があるため。「アプリの予算内に収まらないじゃないか」という不満や、予算不足による治療断念を防ぐ必要があります。
- そのために考えている対策：免責事項の明記とバッファの提案「あくまで平均値である」ことをあらゆる画面で伝えつつ、シミュレーション結果には必ず20%程度の「予備費（お守り代）」を上乗せして表示する。また、ユーザーが自分で項目（保険代など）をカスタマイズできるUIにします。

### 懸念点②
懸念点②：「ゲートキーピング（門番）」による過度な不安
- 何が問題になりそうか：
診断結果が厳しすぎて、本来なら幸せに飼えるはずのユーザーまで「自分には一生無理だ」と諦めてしまう、あるいはアプリを嫌いになってしまうこと。
- なぜそう思うか：
「責任感」を強調しすぎると、完璧主義な人ほど一歩が踏み出せなくなるため。
アプリの目的は「排除」ではなく「準備」であるべきです。
- そのために考えている対策：
「伸びしろ」の提示： 単に「不可」と判定するのではなく、「あと〇〇円貯金があれば安心」「〇〇のグッズを揃えればクリア」といった、「どうすれば飼えるようになるか」という改善策をセットで提示するポジティブなフィードバックを設計します。

■ なぜこの技術を使うのか
Ruby 3.4.x / Rails 7.2.x
最新の安定版を採用することで、開発効率を最大化しつつ、モダンなRailsの機能を活用したいと考えました。特に、今回のような複雑な診断ロジックや計算処理をサーバーサイドで堅牢に実装するのに最適だと判断しました。

Neon (PostgreSQL)
無料枠が充実しており、サーバーレスな特性から、個人開発の初期フェーズにおいて迅速に環境を構築できるため採用しました。

Tailwind CSS
「猫からのフィードバック」といった独自のUIデザインを、CSSファイルを肥大化させることなく、直感的にスピーディに実装するためです。

Chart.js
「一生寄り添う覚悟」を視覚的に伝えるため、生涯費用という重い数字を、ユーザーが受け入れやすい「グラフ」という形に変換するために不可欠だと考えました。

■ 今回チャレンジしたい点
複雑な診断ロジックの構築
単なるアンケートではなく、「生活リズム」「収入」「住環境」といった複数の変数を掛け合わせ、ユーザーごとにパーソナライズされた適切なアドバイスを導き出すアルゴリズムの実装に挑戦します。

JavaScriptを活用した動的なUI体験
学習中のPromiseや非同期処理の知識を活かし、シミュレーション結果をページ遷移なしで瞬時に表示させることで、ストレスのない操作感を実現したいです。

「猫視点」のUX設計
技術的な実装だけでなく、猫のキャラクターを通じたフィードバックなど、ユーザーの感情を動かし「行動変容」を促すためのインターフェース作りにこだわります。

■ 不安な点
計算精度の担保
猫種や年齢、病気のリスクなど、膨大なデータをどのようにDBで管理し、正確な計算式に落とし込むかというデータの構造化に不安があります。

JavaScriptとRailsの連携
Rails側で計算した複雑な生涯コストのデータを、フロントエンドのChart.jsへスムーズに渡し、意図通りに描画させる部分の技術検証が必要です。

初めての外部ストレージ利用
Cloudinaryを用いた画像の動的な管理・表示について、実装の経験が浅いため、ドキュメントを読み込みながら手探りで進める必要があります。

### 10-3.キャッチアップ不足の懸念

キャッチアップ不足の段階では以下の懸念点があります。
1. **完成までのハードルが高い**
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想されます。
2. **Issueの見積もりが難しい**
「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまいます。

例えば、最近注目されているReactやNextについて。
Reactについては**本リリース後に組み込む方向で検討**するのが良いかと考えています。
その後、React単体のキャッチアップを行い、アプリに組み込む技術的な検証をした上で、組み込むかどうかを判断する形が理想かもしれません。

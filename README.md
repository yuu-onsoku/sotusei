# README

## 1. サービス概要
猫を飼う前の不安を診断で解消し、飼う前も飼った後も猫好きの仲間とつながり続けられるコミュニティーアプリです。

## 2. このアイデアはどこから生まれたか
### 2-1. きっかけとなった体験・感情
小学生の頃ねこのテレビ番組や近所の野良猫などを見てねこを飼いたいと思っていましたが、ペット禁止のマンションで飼えないし当時はあまりねこの病院代やお世話代などの金銭面のことやねこの詳しい生体お世話などわかっておらず、かわいいからという安直な理由で飼いたいと思ってしまっていました。

私自身は小学校3年生くらいにはねこを安直にかわいいだけで飼ってはいけないことを学んだが、
テレビやYouTubeなどで、大人になってもかわいいだけで飼おうとしている人がいるのを知りよく無いと思ったことと、「可愛い」という気持ちだけで飼い始めてしまい、多額の通院費や掃除の大変さに直面して困る人や不幸になってしまうねこちゃんを減らしたいと思ったからです。

---

### 2-2. なぜそれが「気になった」のか

- 自分の性格：<br>
  - 心配性　責任感が強い、特にねこに対しては責任を持ちたい

- 過去の経験：<br>
  - 小学生の頃テレビなどでねこが自分勝手な人間に飼われているのを見てかわいそうと思ったから
- 大切にしている価値観：<br>
  - 準備不足によるギャップは、最終的に「飼育放棄」や「猫を捨てる」という悲しい結末を招くことを知ったから少しでも知識を持ってから迎えてほしいという思いから気になりました。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと
本当にねこが飼えるかどうかきちんと考えられるか？
まだ飼ったことがないので飼った後「お医者さんに聞くほどでもないけどこれうちだけ？」といった疑問を聞けない。

---

### 3-2. 本当に解決したい課題は何か
本当に解決したい課題：ユーザーの漠然とした不安を解消して「一生寄り添う覚悟」を決めさせる、あるいは「今はまだ無理だ」という勇気ある決断を促す。
ちょっとしたことでも聞ける場所が欲しい

 - 不安を抱えたり何も調べていないままだと
   - ねこの習性や食べてはいけない物を与えてしまうかもしれない
   - お金がなくて病院やお世話代が払えない
   - ねこのお世話を甘く見てしまい数日家を空けてしまいねこが病気やけがをしてしまう
 - 相談する場所があると
   - 少しのことだったとしても初めて飼う人からすると不安なことが多いので周りに飼ったことがある人がいないと聞きづらいことがある
   - わざわざお医者さんやペットショップに聞きに行く時間や勇気が無いかもしれない

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

年齢設定：
  - 年齢：25〜30代
  - 理由：社会人として自立し始め、生活に癒やしを求めて「本気で飼育を検討し出す時期」で飼いたいけど何から始めたら良いかわからない、身近に相談できる人がいない人に向けて

生活・状況設定：
  - 生活・状況：一人暮らしで寂しくなり元々ねこが好きで初めて猫を迎えようと悩んでいる人。
   - 周りでねこを飼っている人や同じく飼いたい人がおらず相談できない人。
  - 理由：このアプリでねこを飼うには様々な問題があるというのを知ってもらいねこを飼う覚悟を決めてもらうこと。<br>
  また、ちょっとした疑問などを同じく飼いたい人やすでに飼っている人に気軽に聞けて安心できる。

環境設定：
  - 環境：
    - ねこを飼うことに詳しい人が周りにおらず気軽に相談できない
    - ねこ好きが周りにいないので、語れる人がいなくてさみしい
    - ねこを飼いたい意欲はあるが、心配性なため飼うための情報収集をしたい
  - 理由：
    - 飼いたい人と飼っている人達がねこの相談室があるので、ちょっとした疑問でも相談しやすい（メインのコミュニティー機能）
    - 「お迎え診断機能（入口機能）、覚悟のチェックリスト」で、漠然としている費用やお世話などの不安を解消でき、「具体的な準備」に加え、ミスマッチによる飼育放棄を防ぎます。

## 7. このアプリで実現すること

### MVPで作る機能
#### 「猫を飼いたい」という気持ちに対し、「今の生活環境で本当に最後まで責任を持てるか？」を判定する。 ★は特に大切にしている機能<br> 
 - 「お迎え適性」診断機能（入口機能）：
   - 生活リズム（不在時間、帰省頻度）、住環境、経済状況（月々の可動予算）に関する（5〜10問程度の質問）。
   - 回答に基づき「今はまだ早い」「成猫なら可能」「準備万端」などの判定ロジックを実装。
 - 「猫視点」の診断結果表示：
   - 判定結果とともに、猫のキャラクターから「今の生活だとボクたちは寂しいニャ」といった、誠実かつ愛のある理由説明を表示。
- 「覚悟」のチェックリスト機能:
   - 抜け毛、夜鳴き、壁の傷など、猫飼育の「大変な現実」を羅列。すべてに納得したかを確認し、自分の「覚悟」を視覚化します。
 
 - ユーザー登録/ログイン機能：
   - Deviseによる新規登録・ログイン。診断結果をマイページに保存し、後から見返せるようにします。
 - ねこのコミュニティー（メイン機能）：
   - ★ねこの相談室：質問回答機能<br> 
   - ★にゃんスタ：機能（写真投稿機能、いいね｛ねこの肉球｝、コメント投稿機能｛削除機能、ブックマーク「ねこ形本」、タグ｝）
 
---

### 本リリースで作る機能
#### 診断をクリアし、「飼う」と決めたユーザーが、具体的にお迎えの準備を進めるための機能を追加します。
 - （お迎え準備編）：
   - ペットショップ初期費用計算： ねこ全体の、ケージ、トイレ等の初期セット、安心パック（保険等）の合算見積もり。
   - 病院＆ペットホテルアクセス確認： Google Maps APIと連携し、近隣の動物病院を検索・確認する機能。
- Googleアカウント連携：
   - Google Authによる簡単ログイン。
 - RSpecによるテストコード実装：
   - 計算ロジックや診断機能が正しく動くかの自動テスト。
### （できたら）
 - 相談の回答率やベストアンサーの人にバッチを作る
 - パーソナライズ・ロードマップ機能：
   - 診断結果に基づき、「脱走防止柵を買う」「貯金目標○万円」など、その人に必要な準備タスクを自動生成。
 - 準備進捗の可視化（進捗マップ）：
   - 準備タスクの完了度をグラフやステップ図で表示。
   - 「準備完了！」のステータスをSNSにシェアできる機能。
 - 「現実」の費用レコーディング機能：
   - 実際に購入した猫用品や、迎えた後の医療費を家計簿のように記録。

## 8. このアプリの懸念点とその対策
※最低1つは必ず書いてください。

このアプリが**うまくいかない可能性**と、
それに対して**どう向き合おうとしているか**を書いてください。

### 懸念点①
懸念点①：過度責任感の押しつけによる不安
- 何が問題になりそうか：
診断結果が厳しすぎて、本来なら幸せに飼えるはずのユーザーまで「自分には一生無理だ」と諦めてしまう、あるいはアプリを嫌いになってしまうこと。
- なぜそう思うか：
「責任感」を強調しすぎると、完璧主義な人ほど一歩が踏み出せなくなるため。
アプリの目的は「排除」ではなく「準備」であるべきです。
- そのために考えている対策：
「伸びしろ」の提示： 単に「不可」と判定するのではなく、「あと〇〇円貯金があれば安心」「〇〇のグッズを揃えればクリア」といった、「どうすれば飼えるようになるか」という改善策をセットで提示するポジティブなフィードバックを設計します。

■ なぜこの技術を使うのか

Ruby 3.4.x / Rails 7.2.x<br>
最新の安定版を採用することで、開発効率を最大化しつつ、モダンなRailsの機能を活用したいと考えました。特に、今回のような複雑な診断ロジックや計算処理をサーバーサイドで堅牢に実装するのに最適だと判断しました。

PostgreSQL<br>
無料枠が充実しており、サーバーレスな特性から、個人開発の初期フェーズにおいて迅速に環境を構築できるため採用しました。

Tailwind CSS<br>
「猫からのフィードバック」といった独自のUIデザインを、CSSファイルを肥大化させることなく、直感的にスピーディに実装するためです。

Chart.js<br>
「一生寄り添う覚悟」を視覚的に伝えるため、生涯費用という重い数字を、ユーザーが受け入れやすい「グラフ」という形に変換するために不可欠だと考えました。

# 今回チャレンジしたい点<br>
- 複雑な診断ロジックの構築<br>
単なるアンケートではなく、「生活リズム」「収入」「住環境」といった複数の変数を掛け合わせ、ユーザーごとにパーソナライズされた適切なアドバイスを導き出すアルゴリズムの実装に挑戦します。

 - JavaScriptを活用した動的なUI体験<br>
学習中のPromiseや非同期処理の知識を活かし、シミュレーション結果をページ遷移なしで瞬時に表示させることで、ストレスのない操作感を実現したいです。

 - 「猫視点」のUX設計<br>
技術的な実装だけでなく、猫のキャラクターを通じたフィードバックなど、ユーザーの感情を動かし「行動変容」を促すためのインターフェース作りにこだわります。

---

### 不安な点
計算精度の担保<br>
 - 猫種や年齢、病気のリスクなど、膨大なデータをどのようにDBで管理し、正確な計算式に落とし込むかというデータの構造化に不安があります。

JavaScriptとRailsの連携<br>
 - Rails側で計算した複雑な生涯コストのデータを、フロントエンドのChart.jsへスムーズに渡し、意図通りに描画させる部分の技術検証が必要です。

初めての外部ストレージ利用<br>
 - Cloudinaryを用いた画像の動的な管理・表示について、実装の経験が浅いため、ドキュメントを読み込みながら手探りで進める必要があります。

キャッチアップ不足の段階では以下の懸念点があります。
1. **完成までのハードルが高い**
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想されます。
2. **Issueの見積もりが難しい**
計画するのが苦手なので「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまいます。
3. **技術のキャッチアップをどうすればいいのか**
あまり技術についての理解ができていないことと、「何がわからないのかわからない」のできちんとできるのか不安です。

例えば、最近注目されているReactやNextについて。
Reactについては**本リリース後に組み込む方向で検討**するのが良いかと考えています。
その後、React単体のキャッチアップを行い、アプリに組み込む技術的な検証をした上で、組み込むかどうかを判断する形が理想かもしれません。
